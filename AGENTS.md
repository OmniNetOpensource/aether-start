# Repository Guidelines

## Project Overview

`aether-start` is a TanStack Start app deployed to Cloudflare Workers. The product is centered on authenticated AI chat and currently includes:

- streaming chat over Cloudflare Agents / WebSocket
- multi-provider model selection and prompt presets
- conversation persistence in Cloudflare D1
- image attachment storage in Cloudflare R2
- share snapshots, notes, prompt quota, and Better Auth flows
- artifact generation and preview for HTML / React outputs

The codebase has already been split by feature, but the import aliases still preserve older `@/server/*`, `@/lib/*`, and `@/types/*` paths. When a path looks centralized, check `tsconfig.json` before assuming the physical file location.

## Project Structure & Module Organization

- `src/routes/` contains TanStack Start file-based routes. Dynamic segments use `$`.
- `src/routes/__root.tsx` wires the root shell, theme loader, responsive provider, tooltip provider, toast container, and Sentry boundary.
- `src/routes/app/route.tsx` is the authenticated app shell. Keep its login redirect behavior intact.
- `src/routes/app/c/$conversationId.tsx` renders the conversation page with `MessageList`, `Composer`, and the artifact side panel.
- `src/routes/share/$token.tsx` and `src/routes/api/share-assets/$token/$attachmentId.ts` implement public read-only sharing.
- `src/routes/api/assets/$key.ts` serves private R2 assets. `src/routes/api/auth/$.ts` is the Better Auth entry.
- `src/features/chat/` owns chat UI, stores, server agents, tool execution, provider adapters, and server functions.
- `src/features/sidebar/` owns conversation list UI, message-tree helpers, loader hooks, session store, D1 conversation helpers, and conversation server functions.
- `src/features/share/` owns share dialogs, public snapshot types, share DB helpers, and share server functions.
- `src/features/notes/`, `src/features/quota/`, and `src/features/auth/` each keep their own server code near the feature.
- `src/shared/` contains shared UI primitives, shared hooks, shared stores, shared server utilities, env loading, and app-wide helpers.
- `migrations/` contains D1 schema history, including the latest `0015_conversation_artifacts.sql`.
- `src/routeTree.gen.ts` is generated by TanStack Router. Do not edit it manually.

## Path Alias Reality Check

The repo now uses feature-local physical files with compatibility aliases. Common examples:

- `@/server/env` -> `src/shared/server/env.ts`
- `@/server/db/conversations-db` -> `src/features/sidebar/server/conversations-db.ts`
- `@/server/functions/conversations` -> `src/features/sidebar/server/conversations.ts`
- `@/server/functions/chat/*` -> `src/features/chat/server/functions/*`
- `@/types/chat-api` -> `src/features/chat/types/chat-api.ts`
- `@/types/conversation` -> `src/features/sidebar/types/conversation.ts`

If you cannot find a file at the import path, read `tsconfig.json` instead of creating a duplicate module.

## Build, Test, and Development Commands

Use `pnpm` for all tasks.

- `pnpm install` - install dependencies
- `pnpm dev` - start local dev server on port 3000
- `pnpm build` - build production bundle
- `pnpm lint` - run ESLint
- `pnpm type-check` - run TypeScript without emit
- `pnpm check` - run type-check, lint, and build
- `pnpm test` - run Vitest once
- `pnpm test:watch` - run Vitest in watch mode
- `pnpm test:coverage` - run coverage
- `pnpm test:e2e` - run Playwright tests
- `pnpm auth:generate` - regenerate Better Auth schema output
- `pnpm auth:migration:generate` - generate Better Auth migrations
- `pnpm cf:typegen` - regenerate Cloudflare worker typings
- `pnpm cf:migrate:local` - apply local D1 migrations
- `pnpm cf:migrate:remote` - apply remote D1 migrations
- `pnpm cf:deploy` - build and deploy to Cloudflare Workers
- `pnpm cf:sync-secrets` - sync env values to Cloudflare

## Architecture Notes

### Chat Request Lifecycle

- Client request state starts in `src/features/chat/store/useChatRequestStore.ts`.
- Client orchestration lives in `src/features/chat/lib/api/chat-orchestrator.ts`.
- Stream event application lives in `src/features/chat/lib/api/event-handlers.ts`.
- The long-running worker agent lives in `src/features/chat/server/agents/chat-agent.ts`.

Changes to request lifecycle, reconnection, abort handling, or persistence usually need all four checked together.

### Conversation Session State

- `src/features/sidebar/store/useChatSessionStore.ts` is the main client session store.
- It owns message-tree state, conversation list state, current conversation id, model selection, prompt selection, and artifact panel state.
- `src/features/sidebar/hooks/useConversationLoader.ts` hydrates route state from the server and syncs it into `useChatSessionStore`.
- Message-tree helpers live in `src/features/sidebar/lib/tree/`.

If a conversation page bug looks like ŌĆ£route state and store state disagreeŌĆØ, inspect `useConversationLoader.ts`, `useChatSessionStore.ts`, and `src/routes/app/c/$conversationId.tsx` together.

### Artifacts

- Artifact UI lives in `src/features/chat/components/artifact/`.
- The conversation route mounts `ArtifactPanel` beside the chat UI.
- The model can call the `render` tool from `src/features/chat/server/agents/tools/render.ts`.
- Artifact stream assembly lives in `src/features/chat/server/agents/services/render-artifact-stream.ts`.
- Artifact events are part of `src/features/chat/types/chat-event-types.ts`.
- Conversation artifact persistence lives in `src/features/sidebar/server/conversations-db.ts`.
- D1 storage for artifacts was added in `migrations/0015_conversation_artifacts.sql`.

Artifacts are a first-class part of conversation loading now. If you change conversation detail loading or deletion, keep artifact hydration and cleanup in sync.

### Models, Prompts, Providers, and Backends

- Model and prompt definitions live in `src/features/chat/server/agents/services/model-provider-config.ts`.
- Availability APIs live in `src/features/chat/server/functions/models.ts`.
- `src/features/chat/server/functions/roles.ts` is only a deprecated alias.
- Provider adapters live in `src/features/chat/server/agents/services/`.
- `provider-factory.ts` currently supports `anthropic`, `openai`, `openai-responses`, and `gemini`.

Current backends include:

- `rightcode-claude`
- `rightcode-claude-sale`
- `rightcode-gemini`
- `rightcode-openai`
- `dmx`
- `ikun`
- `ikun-gemini`
- `openrouter`

If you add a model, prompt, backend, or provider format, update `model-provider-config.ts` and the related adapter / factory code together.

### Tools

Tool execution lives in `src/features/chat/server/agents/tools/`.

Current tools:

- `fetch_url`
- `render`
- `search` when `SERP_API_KEY` is available

Important files:

- `executor.ts`
- `fetch.ts`
- `render.ts`
- `search.ts`
- `types.ts`

`fetch_url` also depends on `JINA_API_KEY` and `SUPADATA_API_KEY` for some fetch modes.

### Persistence and Server Functions

Server functions are feature-local even when imported through `@/server/functions/*`.

Main areas:

- conversations: `src/features/sidebar/server/conversations.ts`
- notes: `src/features/notes/server/notes.ts`
- shares: `src/features/share/server/shares.ts`
- quota: `src/features/quota/server/quota.ts`
- auth: `src/features/auth/server/*`
- chat helpers: `src/features/chat/server/functions/*`
- shared theme/env helpers: `src/shared/server/*`

Database helpers are also feature-local:

- conversations: `src/features/sidebar/server/conversations-db.ts`
- shares: `src/features/share/server/conversation-shares-db.ts`
- notes: `src/features/notes/server/notes-db.ts`
- quota: `src/features/quota/server/prompt-quota-db.ts`

Worker bindings are resolved in `src/shared/server/env.ts` via `getServerBindings()`.

### Auth

Auth uses Better Auth with Drizzle on D1.

Important files:

- `src/features/auth/server/auth.ts`
- `src/features/auth/server/session.ts`
- `src/features/auth/server/session-state.ts`
- `src/routes/api/auth/$.ts`

Current behavior includes:

- email/password auth
- email OTP plugin
- email verification
- password reset
- IP capture and login timestamp updates
- route protection for `/app`

### Sharing

Sharing is snapshot-based and read-only.

- share DB helpers live in `src/features/share/server/conversation-shares-db.ts`
- share server functions live in `src/features/share/server/shares.ts`
- public rendering lives in `src/routes/share/$token.tsx`

Public shares should stay noindex and must never mutate live conversations.

## Testing Guidance

- Unit and component tests use Vitest with `happy-dom`; config is in `vitest.config.ts`.
- E2E tests use Playwright; config is in `playwright.config.ts`, tests live in `tests/e2e`.
- When changing chat orchestration, provider adapters, tools, route loaders, or DB helpers, update nearby `*.test.ts` / `*.test.tsx` files.
- Recent high-value tests live near:
  - `src/features/chat/server/agents/services/render-artifact-stream.test.ts`
  - `src/features/sidebar/server/conversations-db.artifacts.test.ts`
  - `src/routes/app/c/-$conversationId.test.tsx`

- Local secrets typically live in `.env.local`.
- App env loading is centralized in `src/shared/server/env.ts`.

## Coding Style & Naming Conventions

- TypeScript + React.
- Follow the existing file-local style; most actively edited files use 2-space indentation and single quotes, though some newer files still use double quotes.
- Components use PascalCase, hooks use `useX`, stores use `*Store`.
- Route files mirror their route paths.
- Do not introduce `useMemo`, `useCallback`, or `React.memo`.
- Prefer existing feature/store/server-function boundaries over creating a new abstraction layer.
- Keep changes ESLint-clean.

## Generated Files

- Do not edit `src/routeTree.gen.ts` manually.
- Do not hand-edit generated Cloudflare typings such as `worker-configuration.d.ts` unless you are intentionally regenerating them.
- Treat `src/features/auth/server/auth.schema.ts` as generated output unless the task is specifically about auth generation.
